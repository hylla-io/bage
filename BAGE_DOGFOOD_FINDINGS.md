# Båge Dogfood Findings

Running log of friction, gaps, and bugs found by **using Båge to edit Båge** (see the
[Dogfooding](README.md#dogfooding) policy). Each finding is fixed test-first; when a fix
sharpens one language it lands with test parity across the other file types.

Status legend: **FIXED** · **OPEN** · **WONTFIX** (deliberate constraint).

## Findings

| # | Finding | Surfaced by | Status | Resolution |
|---|---------|-------------|--------|------------|
| 1 | `bage apply --line N` consumed the line's trailing newline, so a `--text` without a newline merged the line into the next one (a blank line in README vanished). | Editing README status line through bage. | **FIXED** | Line mode is now newline-agnostic: `applyRegion` excludes a trailing `\n` from the resolved region and `runApply` strips one trailing `\n` from `--text` in line mode. Covered by `cmd/bage` line-edit tests. |
| 2 | No practical way to apply a multi-line replacement from the CLI without hand-escaping a giant `--text`. | Dogfooding multi-line doc edits (CLAUDE.md, README sections). | **FIXED** | Added `--text-file` to `bage apply`; replacement text is read from a file. Used for every multi-line dogfood edit since. |
| 3 | Båge cannot **read** a file — there is no `show`/`cat` command. | General dogfooding. | **FIXED (show)** | `bage show` (PR #13) ships the addressable read view — blocks + `region_hash` map + file raw/norm hashes, round-trip-verified against `apply`. A raw-bytes `cat` stays absent by design (use the harness reader); the agent-IDE need — see targetable blocks — is met. |
| 4 | Båge cannot **create** a new file — `apply` resolves a region in a file that must already exist. | Creating this file, CONTRIBUTING.md, etc. | **FIXED** | `bage create` (PR #9) creates a new file through the same two-phase/WAL engine, anchored by non-existence (O_EXCL, hard-reject-on-clobber) with a parse floor; also `Editor.Create` on `pkg/bage` and usable inside a mixed-op `ApplyBatch`. |
| 5 | No first-class **append / insert-at-EOF**: adding content to a file's end (this session: SPEC §10, CONTEXT terms, README) requires `grep`-ing the last line's byte offset and replacing `[lastline:end]`, because `apply` needs a non-empty existing region to anchor. | Dogfooding the file-lifecycle design docs through bage. | **OPEN** | Candidate: a zero-width insert (`--start N --end N`) or `--at-end` mode, validated as a pure insertion (no region content to hash — anchor on the file `raw_hash` + offset instead). Fold into the §10 `Op` work. |
| 6 | LSP `rename` returned empty/partial for every non-gopls server: rust-analyzer answered "no references", pyright/clangd edited only the opened file. | E2E rename matrix (rust/python/c++). | **FIXED** | Root cause: `lsp.Client.Rename` did `didOpen`→`rename` immediately with no readiness wait, so a still-indexing server (rust-analyzer) answered before its index existed. The retry already lived in the *test* harness (`container_test.go`), masking the product gap. Added a bounded retry-until-ready loop to `client.Rename` (`renameDeadline`/`renameRetry`, defaults 30s/300ms; `workspaceEditHasChanges` treats an empty edit as not-ready). Verified live (rust-analyzer 3-site rename) + `mage lsp` 11/11. Cross-file *completeness* for pyright/clangd (open referencing files / `compile_commands.json`) is a separate, larger limitation — tracked, not fixed here. |
| 7 | `bage create`/`apply` parse floor **accepts syntactically broken content** (Go `func f( {` was written, exit 0). tree-sitter is error-tolerant — it returns a tree with ERROR nodes rather than failing, so the floor ("did it produce a tree?") always passes. | E2E diagnose run. | **RESOLVED (lenient by design)** | Decided lenient — the floor accepts ERROR/MISSING-node trees so an agent can write broken intermediate states mid-refactor; `diagnose` surfaces the defects and Hylla owns project correctness. Only bytes that produce no tree at all are rejected. Documented in the SPEC gate-boundary note + ADR-0004. |
| 8 | `bage delete` left the now-empty parent dir (`diag/`) on disk; the ADR file-lifecycle section says delete reaps empty dirs. | E2E delete run. | **RESOLVED (won't reap)** | Decided — `delete` does not prune directories; a now-empty parent is left for the host/VCS. Amended ADR-0004 + SPEC §10.5 to drop the reap claim. |
| 9 | `read`/`show` outline is **empty (blocks=0) for data-format grammars** (JSON, YAML; likely TOML/XML/HTML/CSS): `isDeclKind` matches only code declaration kinds, so object pairs / mappings surface no named blocks. | E2E read across file types (v0.3). | **OPEN (enhancement)** | Line/byte addressing + whole-file raw/norm hashing still work on these files (verified: `read --line` and `--start..--end` return anchored `range` blocks), so they are usable — just without named-symbol addressing. Follow-up: extend the Outline declaration-kind set per data grammar (e.g. JSON `pair`, YAML `block_mapping_pair`) so keys/sections become addressable blocks. |
| 10 | `bage read --lines A-B --content` with the default **text** format prints only the block metadata line — the requested content never appears; `--format json` (or `toon`) is required to actually see the bytes. | Porting `internal/session` to Rust: tried to inspect a function body via `read --lines 398-415 --content` before editing it. | **OPEN** | Make the text renderer honour `--content` for range/line reads (print the block content under the metadata line), or reject the flag combination loudly instead of silently dropping the content. |
| 11 | Go `--format toon` prints NOTHING for error envelopes: `session.Kind` is a named string type and toon-go has no `reflect.String` case in `normalize`, so `MarshalTOON` errors and the CLI discards it (`_ = render.Emit(...)`). | Rust rewrite: byte-matching TOON envelopes against the Go binary. | **OPEN** | Fix on the Go side: marshal `Kind` as a plain string (or teach toon-go named string types) and stop discarding `render.Emit` errors. The Rust CLI renders these envelopes correctly. |

## How to add a finding

1. Reproduce the friction while editing a real file through `bage apply` / `bage rename`.
2. Add a row above with what broke and how it surfaced.
3. Fix it **test-first** (mage gates green); if it touches normalize/parse, run `mage fuzz`.
4. Keep test parity across file types — a Go-found fix lands with equivalent coverage for
   the other languages/file types it affects.
5. Flip the row to **FIXED** with the resolution, in the same PR as the fix where possible.
